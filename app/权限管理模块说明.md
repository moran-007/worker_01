# 权限管理模块说明

## 当前目标

权限管理采用“角色分级 + 权限点”的方式实现。超级管理员负责创建账号、分配角色；系统后端按接口端点强制校验权限，前端菜单和按钮只做入口隐藏，不能替代后端拦截。

本阶段先落地主要校区人员：超级管理员、校长、学管师、行政、教师、只读账号。总部、财务、课程顾问等岗位先进入预留文档，等多校区、财务、招生模块补齐后再细化实现。

## 已实现角色

| 角色 | 等级 | 定位 | 当前权限范围 |
| --- | ---: | --- | --- |
| 超级管理员 | 100 | 系统最高权限 | 账号权限、权限分配、全业务管理、日志、数据库关系、备份创建/下载/恢复 |
| 校长 | 80 | 校区负责人 | 校区业务管理、教师/课程/班级/课次/签到/统计、日志查看导出、备份创建下载；不能分配权限和恢复数据库 |
| 学管师 | 60 | 教学运营负责人 | 学生、班级、名单、排课、签到、课程详情、预设课程、统计导出；不能管理账号、数据库、备份恢复 |
| 行政 | 50 | 前台/行政支持 | 学生资料、班级名单、签到协助、常用查看和统计导出；不维护教师、课程体系和排课规则 |
| 教师 | 30 | 授课人员 | 查看本人相关数据，维护本人课次签到和课程详情，查看统计 |
| 只读账号 | 10 | 审阅/临时查看 | 仅查看常用业务数据，不允许新增、编辑、导入、删除、恢复 |

旧账号兼容：

| 旧角色 | 新语义 |
| --- | --- |
| admin | 超级管理员 |
| staff | 学管师 |

旧数据不会被强制迁移；登录和权限判断会自动按新语义兼容。新建或编辑账号时会保存为新的规范角色。

## 核心权限点

权限点集中维护在 `permissions.py` 的 `PERMISSION_CATALOG` 与 `ROLE_PERMISSIONS`。

## 后端实现规则

1. 所有页面和 API 登录后都会进入 `before_request`。
2. `permissions.py` 中的 `ENDPOINT_PERMISSIONS` 定义“端点 + 请求方法 -> 权限点”。
3. 后端按当前用户角色计算权限点，不满足则直接返回 403。
4. 教师的数据范围仍由原有 `apply_teacher_scope`、`ensure_lesson_access`、`ensure_class_access` 负责，做到“有权限也只能看/改自己范围内的数据”。
5. 只读账号对所有非 GET/HEAD/OPTIONS 请求统一拦截。

## 前端实现规则

1. `/api/me` 返回当前用户的 `permissions`、规范角色、角色标签。
2. Vue 路由通过 `meta.permission` 做页面级拦截。
3. 菜单、按钮、表单编辑态通过 `hasPermission(user, permission)` 控制显示或禁用。
4. 账号权限页通过 `/api/permissions` 获取角色列表和权限点，新增/编辑账号时由超级管理员选择角色，并展示权限预览。

## 预留角色

| 预留角色 | 后续方向 |
| --- | --- |
| 总部 | 跨校区监管、总部审计、经营统计、多校区制度配置 |
| 财务 | 收款、退费、课消核算、财务导出 |
| 课程顾问 | 线索、试听、转化跟进、招生报表 |

这些角色已进入权限规划文档，但本阶段不出现在可分配角色里，避免业务模块未完成时出现半可用权限。

## 后续扩展安排

1. 增加新模块时，先在 `PERMISSION_CATALOG` 新增权限点。
2. 再把权限点加入对应角色的 `ROLE_PERMISSIONS`。
3. 新 API 或页面必须加入 `ENDPOINT_PERMISSIONS`。
4. 前端新路由设置 `meta.permission`，按钮级操作使用 `hasPermission`。
5. 如果需要自定义到个人或门店级权限，再新增 `user_permissions` 或 `campus_permissions` 表，保持角色权限作为默认模板。
